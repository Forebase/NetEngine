What is it

A bounded internet you control end-to-end

Not a network simulator. Not an API mock. NetEngines runs real DNS, real TLS, real SMTP, real OIDC — inside a world where you own every root of trust.

🌐

Own the roots

Your DNS root isn't IANA's. Your CA isn't Let's Encrypt. Your identity trust anchor isn't Google. Every chain of trust originates inside the world you control.

⚙️

Real software, real protocols

CoreDNS really serves DNS. Postfix really speaks SMTP. step-ca really issues TLS certs via ACME. Nothing is mocked. It's enclosure, not emulation.

🏢

Organisations as inhabitants

Orgs live inside a world. They own domains, hold address allocations, run mail, host services. They experience the internet from the inside — with all its friction and realism.

The right analogy: LocalStack is to AWS as NetEngines is to the internet's service layer — except LocalStack mocks APIs. NetEngines encloses real ones inside a world where you own all the authority. Enclosure, not emulation. That's a fundamentally different engineering posture.

Architecture

Three reality levels

Everything in NetEngines sits at one of three levels. The levels don't share trust anchors or naming — which is the whole point.

L0

Host Substrate

The real machine: kernel, Docker daemon, orchestrator, physical or virtual hardware. NetEngines runs on this. Given and fixed.

↓
L1

Platform / Engine

NetEngines itself. Holds the declarative spec, wires topology, instantiates the world. Has god-view. Split internally into definition (the YAML spec) and runtime (what executes it).

↓
L1·Org

Organisation

Admitted to the world by the World Registry. Owns resources — domains, address allocations, services — within L2. Sits between platform and in-world: governed by L1 authority, inhabits L2 space. Want full isolation between two orgs? Give each its own world.

↓
L2

In-World

The inhabited internet. Users, mail servers, web services, DNS zones, TLS certificates. Nothing here knows it's enclosed — from inside, the root DNS is the root. Data/control/management planes recur here.

Four cross-cutting concerns

Not levels or planes — properties and vantages that cut across the whole stack.

Operator vantage

The human moves between levels — god-view at L1, org admin at L2 management plane, end user at L2 data plane. A projection, not a fixed layer.

Gateway portal

The single boundary object that straddles L0 and L2. Where the world touches reality, or where two worlds touch each other. Special because it's a boundary, not a service.

Dual identity roots

Two separate trust anchors that must never share a root. L1 identity: who may operate NetEngines. L2 identity: who holds a mailbox, owns a domain, is a user inside the world.

State lifecycle

Ephemeral or persistent. The primary control surface — drives volume strategy, PKI posture, and HA topology. Same spec, same runtime, one sharp threshold to cross deliberately.

Registries

Two registries, two different questions

These are not the same object. Collapsing them produces a god-object with mixed failure modes and mixed governance semantics.

L1-adjacent · World Registry

Who may exist in this world?

Organisations and their capability grants
AND profile assignment per org
World operators and their roles
WHOIS (governance view)

Deleting a World Registry entry removes an entire organisation. This is a governance operation.

L2 · Domain Registry

Who owns what resources?

TLD delegations and governance
Address space — allocated to AND pools
Domain name registrations
Reverse DNS zones

Deleting a Domain Registry entry removes a name or address block. This is a resource operation.

Address allocation model: The Domain Registry allocates address blocks. ANDs consume what they've been assigned — they do not declare their own addresses. The same way a real ISP receives allocation from ARIN/RIPE rather than inventing its own.

Network Model

Administrative Network Domains

Not ISPs. Not VLANs. An AND is an ownership and policy boundary. Each org is assigned a connectivity profile that determines how they experience the network. Addresses come from the Domain Registry.

🏠

Residential

  • Dynamic IP / DHCP
  • NAT enabled
  • Inbound blocked
  • CGNAT experience
  • No reverse DNS
🏢

Business

  • Static allocation
  • NAT optional
  • Inbound allowed
  • Reverse DNS
  • DHCP available
🖥️

Datacenter

  • Public subnet
  • No NAT
  • Full routing
  • Reverse DNS
  • BGP optional†
🔒

Air-gapped

  • No gateway
  • Isolated segment
  • Internal only
  • No inbound/outbound
  • Cyber range use

† BGP fabric is future scope — an optional module, not core.

Services

World services vs org apps

Not the same thing. World services are infrastructure — part of what makes a world a world. Org apps are things organisations deploy inside their own AND.

World Services

Infrastructure for all inhabitants

Provisioned at Phase 8 as part of bootstrap. The world provides these; orgs consume them.

📬Mail backbone — Postfix, DKIM/SPF/DMARC, world-level SMTP
🗄️Object storage — MinIO S3-compatible, platform + org buckets
Org Apps

Deployed by orgs inside their AND

Deployed on demand. Runtime provisions container, registers DNS, issues ACME cert, wires OIDC.

🐙Gitea — self-hosted git
☁️Nextcloud — file sync
📝WordPress — web publishing
✉️Mailpit — dev mail sink (ephemeral only)

The line: if the world wouldn't function as a world without it, it's a world service. If an org deploys it to serve their own inhabitants, it's an org app. Mail infrastructure is the clearest example — the world provides SMTP for everyone; an org can additionally run their own MX inside their AND.

Gateway Portal

The boundary object

Not a service — a role played by the gateway. The single object that straddles L0 and L2. Two separate concerns: how the world relates to the real internet, and how it relates to other worlds.

Real internet posture

How in-world traffic and naming relates to the actual internet outside.

Mode What it means When to use
isolated No real internet. Only in-world names resolve. Fully self-contained. Default. Security labs, pure dev environments, demos.
shadowed Real internet reachable, but in-world names take priority over real ones. Dev environments that occasionally need real internet access.
mirrored Real hostnames (github.com, s3.amazonaws.com) resolve to in-world services. App thinks it's talking to the real internet. Integration testing without real external dependencies.
exposed In-world services published outward via gateway NAT. The world accepts real inbound traffic. Long-lived testbeds, shared infrastructure accessible from outside.

Cross-world connectivity

How this world relates to other NetEngines worlds.

Mode What it means What's shared
none Fully isolated. No cross-world connectivity. Nothing
peered Routing between worlds. DNS resolution across boundaries. No shared trust. Routes + DNS
federated Active trust establishment — OIDC federation, PKI cross-signing, cross-domain mail with full SPF/DKIM/DMARC between world CAs. Routes + DNS + Identity + PKI

Isolation escape hatch: if two organisations need complete isolation — separate roots of trust, no shared infrastructure, no lateral movement risk — give each its own world and set cross_world: none. That's a cleaner answer than trying to enforce hard walls between orgs inside a shared world.

Lifecycle

One dial, one sharp threshold

Same spec, same runtime. lifecycle: ephemeral | persistent drives volume strategy, PKI posture, and HA topology. Build against ephemeral first. Cross the threshold deliberately.

⚡ Ephemeral

Per-branch, per-CI-run, per-dev. Fast spinup, full teardown.

  • CA keys generated at spinup, discarded on teardown
  • No volume durability required
  • Single instance — no HA needed
  • Reload API applies diffs immediately, no safety guards
  • State loss is expected and fine
  • Mailpit as mail sink — no real SMTP delivery

🔐 Persistent

Long-lived shared testbed. Multi-org. Production-grade posture.

  • PKI key ceremony — durable root CA, rotation calendar
  • Durable volumes with backup strategy
  • HA topology for critical services
  • Reload API refuses dangerous changes
  • Reset requires explicit confirmation
  • Real Postfix SMTP — actual mail delivery mechanics
Implementation

Key components

Every piece is open-source and already exists. The value is the assembly, the declarative surface, and the bootstrap sequencing on top.

📡

DNS

CoreDNS as root and TLD nameservers. Three-tier hierarchy: root → TLD → authoritative. DNSSEC for in-world zones.

🔐

PKI

step-ca with ACME provisioner. Issues certs to itself first to break bootstrap circularity. All services get real TLS.

🪪

Identity

Keycloak with two separate realms — platform (L1) and in-world (L2). Different trust roots, different issuers, never shared.

📬

Mail

Postfix + OpenDKIM. DKIM keys published to DNS at Phase 8. SPF and DMARC per domain. Real SMTP deliverability mechanics.

🗄️

Storage

MinIO for S3-compatible object storage. Platform-level bucket for NetEngines itself; orgs allocated buckets on demand.

🐍

Runtime

Python + FastAPI operator API. Pydantic spec models. Async phase handlers. DAG walk, topological sort, diff-and-reload.
