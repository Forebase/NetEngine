What MVP delivers: netengines up against a minimal spec produces a fully bootstrapped world — root DNS resolving, ACME issuing real certs, two Keycloak realms (platform + in-world), address-allocated ANDs with nftables policy, Mailpit capturing mail, MinIO live, and a Gitea instance inside acme.internal that has a real TLS cert and working SSO. Pre-decisions

Lock these before writing a line of code

Three decisions that propagate everywhere. Getting them wrong at M0 means expensive refactoring at M4–M5. They are not architectural preferences — they affect schema, connection config, handler interfaces, and test fixtures from day one.

DECIDE

Supabase: cloud vs self-hosted for MVP. Cloud is faster to start. Self-hosted is the correct posture for authority-autonomous claim. For dev sandbox MVP, cloud is acceptable — the governance data leaves the world, but that tension only matters for the alternative-internet-substrate archetype. Decision: cloud for MVP, self-hosted in persistent mode scope. This affects the Supabase client connection config, secrets management in .env, and CI fixture setup. Pick before writing any Supabase client code.

LOCK

pgmq event envelope schema. Every handler that emits events must agree on this shape before M4. Retrofitting correlation IDs and parent event IDs across all handlers after the fact is painful. Minimum required fields:

{
  "event_id":       "uuid-v4",         // this event
  "correlation_id": "uuid-v4",         // root of the causal chain (same as event_id for root events)
  "parent_event_id": "uuid-v4 | null", // direct parent in the causal chain
  "event_type":     "domain_registered | dns_update_required | dkim_keys_generated | ...",
  "emitted_by":     "domain_registry_handler",
  "emitted_at":     "ISO-8601",
  "payload":        { /* event-specific data */ },
  "retry_count":    0
}

Every queue that crosses a handler boundary needs a corresponding DLQ. Name convention: {queue_name}_dlq. After N retries (default 3), the message moves to the DLQ and the operator API must surface it. Decide N and the retry backoff strategy at M4 before any queue is written.

LOCK

Gateway handler interface contract. The AND handler calls the gateway handler. When VyOS replaces Alpine+nftables for the BGP scope, that swap must be a handler implementation change, not a call-site refactor. Define the interface now, even though only the nftables implementation ships:

class BaseGatewayHandler(ABC):
    async def generate_rules(self, and_name: str, profile: str, cidr: str) -> str:
        """Generate ruleset string for this AND's profile."""

    async def apply_rules(self, and_name: str, rules: str) -> None:
        """Write rules to gateway and reload atomically."""

    async def remove_and(self, and_name: str) -> None:
        """Remove all rules for a deprovisioned AND."""

    async def reload(self) -> None:
        """Full ruleset reload (e.g. after gateway restart)."""
Critical path

Dependency order — milestones are sequential

Each milestone has hard pre-requisites from the one before it. The bootstrap dependency graph is real — phases cannot be parallelised when earlier phase outputs are runtime facts the next phase consumes.

M0 Foundation
→
M1 Substrate + DNS
→
M2 PKI
→
M3 Platform identity
↓
M4 Registries
→
M5 In-world identity
→
M6 ANDs
↓
M7 Services + apps
→
M8 Operator API + reload

There is no parallel track. DNS must resolve before PKI bootstraps. PKI must issue before Keycloak can get a cert. Keycloak platform realm must exist before the operator API can validate tokens. World Registry must be seeded before in-world identity has realms to provision. The sequence is the implementation. Do not attempt to work multiple milestones in parallel during initial build — you will spend more time on integration than you save on calendar time.

Testing strategy per milestone: each milestone ends when a working integration test passes end-to-end for that milestone's scope. Don't accumulate untested milestones. The bootstrap is a causal chain — a broken Phase 1 DNS output breaks every healthcheck downstream silently. Test before proceeding.

The circularity break — what actually happens

Phase 2 provides DNS. But step-ca needs TLS to serve ACME. And ACME issues TLS certs. Neither can start without the other.
Resolution: step-ca starts on a hardcoded IP, not a DNS name. It uses the root CA private key to sign its own TLS cert directly — no ACME request needed.
From this point: ACME directory is at https://10.0.0.6/acme/acme/directory. Any client that trusts the CA cert (installed in trust store) can request a cert.
The IP is only needed for bootstrap. After ca.platform.internal is registered in DNS, all downstream consumers use the hostname.
Org app cert issuance: the app requests certs on containers' behalf via step-ca admin API and injects via volume mount — containers do not run their own ACME client.

M6
ANDs — Administrative Network Domains
Phase 7
Per-org isolated Docker bridges, address allocation from Domain Registry pools, nftables policy rules generated from AND profiles and applied atomically to the gateway. After M6: network segmentation is structural, not rule-based — acme-corp cannot reach bob-home-net at L2, and the residential AND truly has NAT blocking inbound. This is the most operationally complex milestone.

AND provisioning sequence

Create isolated Docker bridge per AND — --internal flag prevents Docker NAT attachment
Address sub-range acquired from Supabase address_pools via row-level lock
Lease written to address_leases — idempotent (same AND always gets same allocation)
Gateway container attached to the new bridge — docker network connect with specific IP
nftables rules generated from AND profile via gateway_handler.generate_rules()
Rules applied atomically via docker exec nft -f /etc/nftables/rules.nft
DNS suffix zone registered via pgmq → dns_handler
and_handler — coordinates docker, gateway, dns, Supabase in correct order

nftables rules per profile

Residential: NAT masquerade outbound, drop unsolicited inbound, no reverse DNS
Business: stateful accept inbound, reverse DNS, no lateral movement between ANDs
Datacenter: full inbound, no NAT, optional BGP stub (nftables passthrough)
Air-gapped: drop-all on gateway interface — no inbound, no outbound
Rule correctness test: generate rules → parse with nft --check → assert semantic intent
Rule reload test: change AND profile via operator API, verify new rules loaded without connection loss
gateway_handler implements BaseGatewayHandler ABC (locked at M0) — nftables impl only

Definition of done

Container in acme-corp-net can reach core bridge (business profile); container in bob-home-net cannot receive unsolicited inbound (residential profile — verify with nft counters)
acme.internal DNS suffix resolves from inside acme-corp-net
Address allocation is deterministic — tearing down and re-running assigns the same CIDR to the same AND
Profile change via operator API applies new nftables rules within 5s without disrupting existing connections

Explicitly not in M6

VyOS / BGP — nftables only
Multiple ANDs per org
DHCP server per AND (spec has dhcp: true in profiles — defer real DHCP daemon; static allocation is sufficient for MVP)
Cross-AND routing policy beyond gateway choke point

Diff / reload engine

Immutability guard runs first — reject entire reload if any immutable field changed
Diff: old spec vs new spec → added / removed / updated per section
Apply in dependency order — topological sort of change set
Ephemeral: apply immediately, no safety guards
Partial failure: halt at failing step, log completed + failed; re-POST is safe (idempotent handlers)
Handler idempotency: create network that exists → return existing; zone record exists → update not duplicate

Observability

Structured phase logs queryable by phase number and timestamp via API
Phase logs streamed to CLI via Supabase Realtime during bootstrap
CLI: netengines up, netengines reload, netengines status, netengines down
Pretty-print phase progress in CLI — phase status, healthcheck results, errors

netengines up single-org.yaml completes all 9 phases with all healthchecks green
Add an org to the spec, netengines reload — new org admitted, AND provisioned, in-world identity created, no world reset required
