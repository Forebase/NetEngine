How NetEngines
actually works

The briefing covers what and why. This document covers how — implementation choices, sequencing mechanics, topology wiring, and where the hard problems live.

App model

Controller, not absorber

The NetEngines Python app is a controller and coordinator. It does not replace in-world services — it orchestrates them. CoreDNS, step-ca, Keycloak, Postfix, MinIO remain what they are. The app calls them via their native interfaces and uses Supabase to coordinate sequencing, persist state, and propagate events between handlers.

L1 app

NetEngines operator surface

REST API, CLI, phase orchestrator, diff engine, handler coordination. Authenticated via L1 identity. May have multiple surfaces — operator API (admin), management plane API (per-org).

↕
backend

Supabase

Postgres (runtime state, world registry, address allocation, operator records), pgmq (inter-handler event queue, async ops, retry), Realtime (push to connected operator clients), Storage / Auth as needed.

↕
Docker

Container runtime (L0 substrate)

Docker SDK. App creates networks, starts containers, attaches volumes, runs exec commands. One isolated bridge per AND. Gateway container multi-homed across all AND bridges.

↕
L2

In-world services

CoreDNS, step-ca, Keycloak (in-world realm), Postfix/OpenDKIM, MinIO, gateway container. Real software, real protocols, real in-world IPs. The app seeds and configures them — it does not replace them.

The line: if in-world inhabitants or standard tooling needs to speak a protocol to it (DNS on port 53, SMTP on port 25, WHOIS on port 43, ACME over HTTPS), it lives in-world as a real service. If it is orchestration, governance, or state management that only the app needs, it lives in Supabase and is surfaced via the operator API.

Versioned surfaces

The app may expose distinct surfaces per layer. At minimum: an L1 operator API (admin control, world management, phase orchestration) and potentially an L2 management plane API (per-org self-service — register a domain, deploy an app, provision a mailbox). These are separate auth boundaries and separate concerns. Not required for MVP — start with the operator API only.

⚙️

Operator API — L1

World management, phase orchestration, org admission, reload. Authenticated via L1 identity (Keycloak platform realm). Listens on platform network only — not routable from in-world.

🏢

Management API — L2 (future)

Per-org self-service. Deploy an app, register a domain, provision users. Authenticated via L2 identity (in-world Keycloak realm). Lives at a routable in-world address.

Supabase backend

Full backend for the app layer

Supabase is what the NetEngines app needs — not a replacement for in-world services. Postgres for persistence, pgmq for queuing, Realtime for push, Auth and Storage available where needed. Self-hosted for the authority-autonomous posture to hold.

postgres
Runtime state + world registry

Phase completion, handler outputs, org admission records, AND assignments, address pool allocations. Row-level locking on address assignment prevents IPAM conflicts.

pgmq
Inter-handler event queue

Domain registration triggers DNS zone update. Phase completion triggers downstream phase. Async ops with retry. Replaces the event bus problem cleanly — it's a queue message, not a custom pub/sub.

realtime
Operator push

Live phase progress to connected CLI or web clients. Bootstrap log streaming. Service health state changes. No polling needed.

postgres
Operator audit log

Every API call, phase transition, reload diff. Queryable, durable, used for troubleshooting and persistent-mode safety guards.

auth
Platform identity (optional)

Alternative to Keycloak for L1 operator auth — see identity section. Only one should be used. Keycloak is preferred for consistency with spec.

storage
World export / import

Spec snapshots, exported world state tarballs. Persistent-mode backup artefacts. Not a replacement for MinIO in-world — separate concern.

Self-hosted Supabase is the right posture. Cloud-hosted Supabase puts the world's governance data outside the world's infrastructure — a tension with the authority-autonomous claim. For dev sandbox use cases this is acceptable. For the alternative internet substrate archetype it is not. Self-hosted Supabase adds operational weight; treat it as a known cost, not a surprise.

Docker topology

The segmentation problem — and the fix

Docker Swarm and Compose are built to maximise reachability. Overlay networks, service discovery, shared DNS — all designed so containers can find each other. NetEngines needs the opposite: deliberate segmentation, routed boundaries, NAT at the edge. These goals are in direct conflict.

Why a shared overlay fails: if acme-corp-net and bob-home-net both attach to a Swarm overlay for management, they can reach each other directly at L2. There is no router between them. The "inbound blocked" firewall rule for the residential profile has nowhere to live. The broadcast domain boundary that makes a residential AND feel like residential simply does not exist.

The correct topology

Each AND gets an isolated Docker bridge — not a shared overlay. The gateway container is the only container that attaches to more than one bridge. All inter-AND traffic must physically flow through it. This is not a policy choice — it is a topology fact that enforces the choke point.

┌─────────────────────────────────────────────────────────────────┐ │ L0 HOST │ │ │ │ [platform bridge] 172.20.0.0/16 │ │ │ │ NetEngines app · Supabase · Operator API │ │ │ │ [core bridge] 10.0.0.0/8 │ │ │ │ CoreDNS root · step-ca · Keycloak · Postfix · MinIO │ │ │ │ [gateway container] ←── multi-homed: one interface per bridge │ │ │ │ │ ┌───┴──────────────────────────────────┐ │ │ │ │ [AND bridge: acme-corp-net] [AND bridge: bob-home-net] │ 192.168.1.0/24 192.168.2.0/24 │ │ isolated · policy: business │ isolated · policy: residential │ │ inbound: allowed │ inbound: blocked / NAT │ │ reverse DNS: yes │ reverse DNS: no │ │ │ │ [acme containers] [bob containers] └─────────────────────────────────────────────────────────────────┘

What the AND handler actually does

1

Create isolated bridge per AND

Docker SDK: docker network create --driver bridge --internal netengines_and_{name}. The --internal flag prevents the bridge from attaching to the default Docker NAT — isolation is structural, not rule-based.

docker SDK--internal
2

Assign subnet from domain_registry allocation

Query Supabase address_pools for the AND's profile. Acquire a sub-range via row-level lock. Pass it to the bridge creation call. Write the lease to address_leases.

supabasepgmq
3

Attach gateway to the new bridge

docker network connect netengines_and_{name} netengines_gateway --ip {gateway_ip}. The gateway now has an interface on this AND. Inter-AND traffic has a routed path.

docker SDK
4

Generate and load nftables rules

App generates nftables ruleset from AND profile (NAT masquerade for residential, stateful accept for business inbound, drop-all for airgapped). Writes to shared volume. Runs docker exec netengines_gateway nft -f /etc/nftables/rules.nft. Reload is atomic — nft -f applies the whole ruleset in one operation.

nftablesdocker exec
5

Register DNS suffix

Emit a pgmq event → DNS handler adds the AND's dns_suffix zone to CoreDNS and reloads. All containers in the AND resolve via the world's DNS.

pgmqcoreDNS reload

Compose/Swarm scope: Compose and Swarm are only used within a single AND — containers in acme-corp-net can discover each other freely via Docker's built-in DNS. Cross-AND traffic routes through the gateway. Swarm overlay networks are not used for production topology — only for management-plane connectivity between the NetEngines app and L1 services.

Gateway implementation

What actually runs in the gateway container

The briefing describes gateway roles and modes. This section describes what implements them — and what's deferred.

MVP

Lightweight — no external tooling dependency
IP forwarding enabled on container start

Rules generated by app from AND profiles, written to volume
nft -f reload is atomic — no dropped connections

Needed when bgp_fabric module is enabled

nftables rule structure per AND profile

# Generated by app/handlers/and_handler.py → gateway_handler.py
# Written to /etc/nftables/rules/{and_name}.nft on gateway container

table ip netengines_acme_corp {
  chain forward {
    type filter hook forward priority 0;
    # business profile: stateful, inbound allowed
    iifname "eth_core" oifname "eth_acme_corp" ct state established,related accept
    iifname "eth_acme_corp" oifname "eth_core" accept
    iifname "eth_acme_corp" oifname "eth_acme_corp" drop  # no lateral movement
  }
}

table ip netengines_bob_home {
  chain forward { ... }
  chain postrouting {
    type nat hook postrouting priority 100;
    # residential profile: NAT masquerade
    oifname "eth_core" masquerade
  }
  chain prerouting {
    type nat hook prerouting priority -100;
    # residential: block unsolicited inbound
    iifname "eth_core" ct state new drop
  }
}

Rule reload on AND profile change: changing an AND profile via the operator API triggers the AND handler to regenerate the nftables file and call docker exec netengines_gateway nft -f /etc/nftables/rules.nft. Because nft -f loads atomically, existing established connections survive the reload. This is the mechanism for live profile changes in ephemeral mode; persistent mode adds a safety guard requiring explicit confirmation.

Where to test gateway rules — and where not to. The correctness guarantee needs to live in the profile → ruleset translation layer, not in a parallel simulator environment. The right tests: given a profile definition, generate the nftables ruleset, parse it back with nft --check, and assert the semantic intent matches (residential has masquerade rule, business has stateful accept, airgapped has drop-all). This is pure Python unit test territory. A separate gateway simulator adds a second environment to maintain and doesn't actually test the translation logic. Dense unit tests on gateway_handler.generate_rules(profile) is the correct investment.

Registry implementation

What runs, what the app coordinates

The registries are real in-world services — the app seeds them, coordinates interactions between them, and does not absorb them. What backs each one differs by concern.

Concern Lives where Backed by Who calls it
Org existence / capabilities World Registry (in-world service) Supabase world_registry table App as coordinator; in-world services query via operator API
Address allocation Domain Registry (in-world service) Supabase address_pools + address_leases AND handler at Phase 7; row lock prevents conflicts
Domain registration Domain Registrar (in-world service) Supabase domain_records + CoreDNS zone file App or org via registrar service; triggers DNS handoff via pgmq
WHOIS In-world — port 43 TCP listener Queries Supabase domain_records + world_registry Standard WHOIS clients inside the world; protocol: RFC 3912
RDAP (future) In-world — HTTPS endpoint Same Supabase tables as WHOIS Modern registry clients; replaces WHOIS long-term

WHOIS implementation: a thin TCP server (port 43) as a container on 10.0.0.9. Accepts plaintext queries, looks up domain_records and world_registry via the app's internal API, formats and returns RFC 3912 plaintext. Approximately 80 lines of Python. The briefing gives it a dedicated container and IP — that is correct, because in-world tooling needs to reach it on the standard port. The data source is Supabase, but the protocol surface is in-world.

Phase 5 — what actually runs

The spec implies containerised registry services at their IPs. Phase 5 is more precisely: the app seeds Supabase from the spec (orgs, address pools, TLD delegations), starts the WHOIS listener, wires the domain registrar service, and ensures the operator API registry routes are live. The heavy lifting is data seeding, not container orchestration.

DNS handoff

Domain registration → CoreDNS reload

The sequence from an org registering a domain to that domain resolving inside the world. Previously unaddressed — the cross-handler coordination was implicit and untraced.

1

Registration request arrives

Org submits domain registration via the domain registrar service (registrar.platform.internal) or via operator API. App validates org capabilities (must hold register_domains) against world_registry.

operator APIworld_registry check
2

Write to domain_records

App inserts row into Supabase domain_records: name, org, NS records, TTL, created_at. This is the authoritative registration. Atomic — if this fails, nothing downstream happens.

supabase write
3

Enqueue DNS update event

App publishes to pgmq: { event: "domain_registered", name: "acme.internal", org: "acme-corp", ns: [...] }. The DNS handler subscribes. Decoupled — DNS failure does not roll back the registration; it retries the DNS update.

pgmq publish
4

DNS handler generates zone update

DNS handler dequeues the message. Calls dns_handler.add_zone_record() which generates the zone file delta and writes it to CoreDNS's config volume (or calls CoreDNS admin API if zone reload is supported without full restart).

dns_handlerzone file write
5

CoreDNS reload — via reload plugin

DNS handler writes zone file to a watched config directory. CoreDNS's reload plugin with the auto directive detects the change and hot-reloads without restart. docker exec SIGHUP is not the primary mechanism — it creates a coupling to the Docker runtime that breaks under k3s where docker exec doesn't exist. The reload plugin works regardless of orchestrator.

coreDNS reload pluginauto directiveorchestrator-agnostic
6

DKIM record injection (mail domains)

If the registered domain has mail services, a second pgmq event triggers the mail handler to generate DKIM keys, publish the TXT record to CoreDNS, and configure Postfix for the domain. Same sequence as above — DNS side effect of a domain event.

pgmqmail_handlerDKIM TXT → coreDNS

pgmq decouples registration from DNS propagation. The registration is committed first. The DNS update is a side effect that can retry on failure without rolling back the registration. This also means the DNS handler can be tested in isolation — replay events, validate zone output, without touching the domain registry.

Dead-letter queue is not optional. If the dns_handler is down or fails repeatedly, messages must not silently disappear. Every pgmq queue that crosses a handler boundary needs a DLQ. After N retries the message moves to dlq_{queue_name}. The operator API must expose DLQ state — GET /api/v1/queues should show message counts, stuck events, and allow replay. An operator who can't see that "DNS event for acme.internal is stuck in DLQ" has no path to recovery. Treat DLQ visibility as a first-class operator tool, not an afterthought.

ACME bootstrap

Breaking the PKI circularity

step-ca needs TLS to serve ACME. ACME issues TLS certs. Neither can start without the other — unless one side uses an IP address instead of a DNS name for the initial connection. That's the circularity break.

1

Phase 3 starts — DNS and networks exist, PKI does not

Phases 0–2 complete: Docker bridges created, CoreDNS root running on 10.0.0.2, TLD delegation chain functional. DNS resolves in-world names. No certificates exist yet.

phase 0–2 complete
2

Generate CA root key material

App calls pki_handler.generate_root_ca(). In ephemeral mode: generates new key/cert pair, writes to ephemeral volume. In persistent mode: either loads from durable volume (if exists) or runs key ceremony. CA cert PEM is written to runtime_state.ca_cert_pem in Supabase.

pki_handlersupabase write
3

Start step-ca on hardcoded IP

App starts step-ca container on 10.0.0.6 (from spec pki.root_ca.acme.listen_ip). Container has the CA key material via volume mount. step-ca generates a self-signed cert for its own identity using the root CA key — it is its own first client.

docker create10.0.0.6 (hardcoded)
4

ACME directory available via IP

https://10.0.0.6/acme/acme/directory is now reachable — by IP, not DNS name. The healthcheck uses this URL directly. Any downstream Phase 3 client that needs to trust the CA cert must install the root CA cert in its trust store before calling this endpoint.

healthcheck: https://10.0.0.6/acme/acme/directory
5

DNS name registration for ACME server

App adds ca.platform.internal → 10.0.0.6 A record to the platform zone. From this point, downstream services can use DNS-based ACME normally — https://ca.platform.internal/acme/acme/directory. The IP address was only needed to break the initial circle.

dns_handlerca.platform.internal
6

Org app ACME certificate issuance

When an org app container is deployed (Phase 8), the app requests the cert on the container's behalf via the step-ca admin API — not the container running its own ACME client. App writes the cert and key to a shared volume the container mounts. Rationale: simpler container images (no ACME client), app controls cert lifecycle, cert status visible in operator API.

app_handlerstep-ca admin APIvolume mount

Org app deployment sequence: app_handler coordinates four steps in order — (1) start container inside org's AND bridge, (2) emit DNS registration event via pgmq → dns_handler, (3) request cert from step-ca via pki_handler and inject via volume, (4) create OIDC client in org's in-world Keycloak realm via oidc_handler. If any step fails, the partial deployment is logged to operator audit and can be retried or rolled back independently.

pgmq as state machine, not just a queue. When domain_registry_handler emits an event the dns_handler processes, which then triggers mail_handler, which emits back to dns_handler — the chain has observable state. Each message should carry a correlation ID and a parent event ID. The operator API should surface this as a queryable event graph: GET /api/v1/events/{correlation_id} shows the full chain, current state of each step, and where it's stuck. This is the difference between a system you can operate and one you can only restart.

Reload / diff

Spec diffing and live apply

POST /reload computes the delta between old and new spec and applies changes in dependency order. What counts as a detectable change, and what happens on partial failure.

Immutability policy — what cannot change after bootstrap

Certain fields in the spec are facts about the world's physical substrate. Changing them post-bootstrap is not a reload — it is a reset. This policy must be codified in the spec schema itself, not caught at runtime. Marked fields produce a validation error on reload diff, not a runtime failure.

FieldWhy immutableChange path
substrate.networks.* L0 Docker bridges. Every container IP depends on these CIDRs. Changing a subnet means every container in the world has the wrong address. World reset only
dns.root.listen_ip Hardcoded into every container's resolver config at spinup. Changing it silently breaks all DNS in the world. World reset only
pki.root_ca.acme.listen_ip Hardcoded into every service's ACME config and trust store. Changing it breaks all cert issuance. World reset only
metadata.lifecycle Ephemeral → persistent is a PKI key ceremony and storage migration, not a config change. Persistent → ephemeral destroys durable state. Explicit migration procedure
domain_registry.address_space[*].cidr Existing AND leases reference these. Changing a pool CIDR while leases are active produces overlapping or invalid allocations. New pool entry only; existing pools frozen

Immutability must be enforced at spec parse time, not handler time. If the reload diff detects a change to any immutable field, the entire reload must be rejected with a clear error before any handler is called. A partial apply that changes mutable fields but fails on an immutable one leaves the world in an inconsistent state. The validator runs before the orchestrator — no exceptions.

Diff categories

Change typeDetectionEphemeralPersistent
New org added org name in new spec not in world_registry apply apply
Org removed org name in world_registry not in new spec apply refuse — governance operation, requires explicit API call
AND profile changed profile field diff on existing AND instance apply — regenerate nftables confirm — warn before applying
New org app deployment entry not in current state apply apply
PKI reconfiguration any change to pki.* block apply — ephemeral CA, no ceremony refuse — requires explicit key ceremony outside reload
Gateway mode change gateway_portal.real_internet.mode diff apply — regenerate rules confirm

Partial failure handling

Apply runs in dependency order. If a step fails, the apply halts at that step. Steps already applied are not rolled back — the world is in a partially-updated state. The operator audit log records exactly which steps succeeded and which failed. The operator can either fix the underlying issue and re-POST /reload (idempotent where possible) or reset in ephemeral mode.

Idempotency target: handler operations should be idempotent where possible — creating a Docker network that already exists returns the existing network, not an error; CoreDNS zone records that already exist are updated not duplicated. This makes partial-failure recovery via re-applying the full spec safe without manual cleanup.

Debugging tools must be first-class from day one. The system will spend a significant amount of time in partially-bootstrapped states during development and early operation. The operator needs: structured phase logs queryable by phase number and timestamp; event queue state (pending, processing, DLQ counts per queue); healthcheck status per phase accessible via GET /api/v1/health; and a way to replay a stuck event from DLQ without restarting the world. These are not post-MVP additions — a system you cannot inspect is a system you cannot debug, and debugging is most of the work during the bootstrap phase of development.

Identity

Two trust roots — and a decision

The architecture requires two separate identity trust anchors that must never share a root. One open question needs resolution before implementation.

L1 — Platform identity

Who may operate NetEngines

Controls operator API access
Issues tokens for phase orchestration, org admission, reload
Listens on platform network only — not in-world
Decision needed: Keycloak platform realm vs Supabase Auth — pick one, not both
Keycloak preferred — consistent with spec, proper OIDC flows
L2 — In-world identity

Who inhabits the world

Mailboxes, domain ownership, SSO for org apps
Separate Keycloak realm — different issuer URL, different signing key
Routable from in-world — lives at auth.internal
Seeded from world_registry org list at Phase 6
Never shares tokens or trust anchor with L1

The open decision: Supabase Auth exists as a capability. The spec has Keycloak for the platform realm. Using both produces two competing L1 identity systems, which is worse than either alone. The recommendation is Keycloak for L1 — it gives proper OIDC flows, is already in-world as a container, and is consistent with the spec. If Supabase Auth is used for anything, it is a different concern (e.g. app-layer developer accounts on a SaaS offering of NetEngines) — not L1 operator identity.

Bootstrap auth problem

The operator API validates tokens against the L1 Keycloak realm. But Keycloak is itself a Phase 4 bootstrapped service. What authenticates the operator during bootstrap (Phases 0–3)?

Resolution: phases 0–3 run under a bootstrap secret — not unauthenticated. The secret is generated at first run, written to a .env file on the host, and passed as an environment variable to the Python app. The API middleware checks for this secret on pre-Phase-4 endpoints. The CLI reads it from the same .env. It cannot be set by an API call — it is a local host secret that gates bootstrap initiation. After Phase 4 completes, the bootstrap secret is retired for operator API calls; all subsequent requests require OIDC tokens from the platform realm. The secret remains valid only for the CLI's local phase execution path.

Keycloak Admin REST API is legendarily difficult. It is underdocumented, version-sensitive, and occasionally surprising in behaviour. Realm creation, client registration, user provisioning, and role assignment each have their own quirks. The oidc_handler will be one of the most brittle handlers to write and maintain. Mitigations: pin the Keycloak version in the container image and do not auto-update; write integration tests for every Admin API call the handler makes; log the full request and response body on any Admin API error, not just the status code. Budget more time for Phase 4 and 6 than the handler matrix implies.

Archetype tension

Four archetypes, one platform — explicit conflict

These documents are coherent because they implicitly optimise for the dev sandbox archetype. That is the right starting point. But the other three archetypes have requirements that are currently invisible and will surface the moment someone tries to use NetEngines for them. Naming this tension now prevents architecture decisions that close off archetypes unintentionally.

ArchetypePrimary requirementConflicts withCurrent stance
Dev sandbox Fast spinup, ephemeral, mirrored gateway, ACME works, SSO works. Mailpit not Postfix. — optimised for
Teaching lab Protocol fidelity — WHOIS on port 43, real SMTP delivery mechanics, BGP peering students can observe. Multiple isolated student worlds. Reset between cohorts. Dev sandbox deprioritises protocol fidelity (Mailpit, REST WHOIS). BGP is future scope. Multi-world management is unaddressed. deferred
Cyber range Malicious traffic simulation, impersonation of real services, air-gapped ANDs with deliberate attack paths. Forensic-quality logging. Hard isolation between attacker and defender environments. App handler deploys known-good services from a catalog. Injecting malicious services is architecturally outside the current model. The gateway enforces isolation but the spec has no mechanism for deliberate vulnerability injection. deferred
Alternative internet substrate Long-lived, persistent, HA, durable PKI, cross-world federation, real BGP routing fabric, DNSSEC rotation, external exposure. Self-hosted Supabase. Everything that makes dev sandbox fast — ephemeral keys, no HA, Mailpit, cloud Supabase — is wrong for this archetype. The operational threshold is a sharp crossing, not a gradient. persistent mode scope

The immediate risk: architecture decisions made while optimising for dev sandbox may silently close off other archetypes. Specific examples — a REST-only WHOIS endpoint (no port 43 listener) is fine for dev sandbox but breaks teaching lab protocol fidelity. Hardcoding Mailpit as the mail handler for ephemeral mode makes the code simpler but makes it harder to swap in Postfix for the same lifecycle. The AND profile system is extensible — but if the gateway handler only generates nftables rules for the four current profiles, adding a "cyber range attacker" profile later means touching the handler, not just adding a profile definition. Design for extensibility at the boundary points even when MVP doesn't need it.

Cross-world federation is out of MVP scope and should stay that way. BGP, DNSSEC cross-signing, OIDC federation, and cross-CA trust are each individually complex. Combining them for the alternative internet substrate archetype is a later milestone. The architecture supports it in principle (gateway portal cross_world modes, federated mode in spec). Do not let it distract from getting a single world solid first.

Open questions

Unresolved before implementation

Items that need a decision — not captured elsewhere in the briefing or this document.

CLOSED

CoreDNS zone update mechanism: resolved — use the reload plugin with the auto directive. CoreDNS watches the config directory and hot-reloads. docker exec SIGHUP is reserved for emergencies only. Reason: orchestrator-agnostic; works under k3s where docker exec doesn't exist. The dns_handler writes zone files to a watched volume mount and does not call Docker directly for reloads.

CLOSED

L1 platform identity: resolved — Keycloak platform realm. Supabase Auth is not used for operator authentication. If Supabase Auth is used in future it is for a separate concern (SaaS developer accounts) not L1 operator identity. The oidc_handler owns all Keycloak Admin REST API calls for both platform and in-world realms.

DECIDE

Self-hosted vs cloud Supabase for MVP: cloud is faster to start; self-hosted is the correct posture for the authority-autonomous claim. For dev sandbox MVP, cloud is acceptable. For persistent or alternative-internet archetype, self-hosted is required. This choice affects connection config and secrets management from day one — decide before writing any Supabase client code.

DECIDE

Event queue schema: pgmq messages need a correlation ID and parent event ID to support the event graph visibility in the operator API. The message envelope schema must be agreed before any handler emits events — retrofitting correlation IDs across all handlers after the fact is painful. Define the envelope shape first.

CLARIFY

Management API (L2) scope and timing: per-org self-service surface for domain registration, app deployment, user management. The operator API covers all of this for MVP. The L2 management API is needed for the multi-tenancy use case where orgs self-serve without operator involvement. Not needed for MVP — but the auth boundary and routing design should not make it hard to add later. The L2 in-world Keycloak realm already provides the right token issuer; the management API just needs to validate against it.

CLARIFY

Archetype extensibility points: where in the codebase do teaching lab and cyber range requirements plug in without rewriting core handlers? Candidates: AND profile definitions (extensible by design), org app catalog (extensible), gateway_handler rule generation (needs explicit extension point for non-standard profiles). Identify these before MVP code is written, even if the extension points themselves are stubs.

FUTURE

VyOS gateway migration path: when the BGP fabric module comes in scope, the gateway needs to migrate from Alpine+nftables to VyOS. The AND handler should abstract the gateway interaction behind gateway_handler so the migration is a handler swap, not a refactor of every phase that touches the gateway. The interface contract — generate_rules(profile), apply_rules(), reload() — should be defined now even if only the nftables implementation ships.

FUTURE

DNSSEC key rotation: the spec enables DNSSEC with KSK/ZSK rotation schedules. The rotation calendar management — generating new keys, rolling signatures, retiring old keys — is unaddressed. In ephemeral mode this doesn't matter (new keys at spinup). In persistent mode it's a non-trivial operational concern that needs to live in a scheduled job backed by Supabase, with expiry-runway SLOs surfaced in the operator API.
